## Challenges

1. **Navigation Gaps — Features Built Without UI Entry Points** — Feed (`/feed`), Saved Posts (`/saved`), Profile Settings, and Logout are all functional but unreachable from the UI. No links in BottomNav, no profile avatar dropdown, no logout action in AppShell or AdminShell. Story specs focused on feature pages but never included navigation integration. Users cannot discover the features without typing URLs manually.

2. **Dialog Mock-to-Null Pattern Rediscovered 3 Times** (Stories 4.2, 4.3, 4.4) — CSS `md:hidden` doesn't apply in jsdom, so both desktop inline form and mobile Dialog render simultaneously, causing duplicate `data-testid` elements. Fix (mock Dialog to return `null`) was applied in 4.2 but not documented in `vi-patterns.ts`, leading to rediscovery in 4.3 and 4.4.

3. **Route Test Spec Contained Stale Mock Pattern** (Story 4.3) — Spec showed `withApiHandler: (handler) => handler` passthrough, which strips the try/catch that converts `ApiError` to HTTP responses. Tests returned 200 instead of expected 4xx. Required 8 review fixes. Correct pattern: mock `@/lib/rate-limiter` + `@/lib/request-context` dependencies instead.

4. **Drizzle Chain Mock Complexity Escalating** — By Story 4.3, tests needed per-call `mockReturnValueOnce(makeSelectChain(result))` with fresh chain objects because queries had multiple terminal calls. Story 4.4 required `vi.hoisted()` for DB mock objects. Test setup increasingly dwarfs test assertions. Will worsen in Epic 5 with group membership joins.

5. **Dual Feed Page Function Duplication** — `_getChronologicalFeedPage` and `_getAlgorithmicFeedPage` share ~80% identical select clauses. Every `FeedPost` type change (category in 4.2, isBookmarked/pinnedAt in 4.4) required identical updates to both functions. Maintenance risk that compounds with Epic 5 group post visibility.

6. **Accepted Technical Debt (4 items):**
   - `PostRichTextRenderer` creates full Tiptap editor instance per feed item (performance concern at scale)
   - `likeCount` column name is historical artifact — actually represents total reaction count across all 5 types
   - `CommentSection` refetch-on-mutate is simple but wasteful for large comment lists
   - No batch reaction endpoint — 20 individual fetches per feed page for viewer reactions

## Key Technical Decisions & Lessons Learned

- **Application-layer scoring for algorithmic feed**: `db.execute()` with postgres.js returns `RowList` (array-like), NOT `{ rows }`. Application-layer scoring in JS is simpler, testable, and safe at MVP scale.
- **`isBookmarked` via LEFT JOIN**: Both `_getChronologicalFeedPage` and `_getAlgorithmicFeedPage` get LEFT JOIN on `community_post_bookmarks` — no N+1.
- **Route test pattern (CORRECTED)**: NEVER mock `withApiHandler` as passthrough. Mock `@/lib/rate-limiter` (`checkRateLimit → {allowed:true}`) + `@/lib/request-context` (`runWithContext: (_ctx, fn) => fn()`).
- **Dialog mock-to-null in jsdom**: CSS media queries don't apply — mock Dialog/DialogContent to return `null` when testing components with responsive desktop/mobile rendering.
- **`vi.hoisted()` for DB mock objects**: When `vi.mock("@/db", () => ({ db: mockDb }))` references a `const` declared after the `vi.mock()` call, use `vi.hoisted()` to avoid TDZ errors.
- **Asymmetric server action return types**: `reactToPostAction` and `toggleBookmarkAction` return `{ data }` on success (no `success` field). Error detected via `"errorCode" in result`. Contrasts with `createPost` which uses `{ success: true }`. Convention needs formal documentation.
- **Self-referential FK pattern**: `parentCommentId` and `originalPostId` use plain `uuid()` without `.references()` — FK enforced by migration SQL only. Avoids Drizzle circular reference issues.
- **`pinnedAt` ordering**: Chronological feed uses `CASE WHEN isPinned THEN pinnedAt ELSE NULL END DESC NULLS LAST` for "most recently pinned first" ordering.
- **`waitFor` for async rollback**: Optimistic update rollback happens after async server action promise resolves, not synchronously. Always use `await waitFor(...)` for rollback assertions.
- **Drizzle chain mocks per-call**: Use `mockReturnValueOnce(makeSelectChain(result))` with fresh chain objects when query has multiple terminal calls (`.limit()` vs `.orderBy()`).

## Team Agreements

### Carried Forward from Epic 3 (still active):

- i18n keys defined in `en.json` + `ig.json` **before** any UI component is scaffolded
- `vi.useRealTimers()` required for tests involving React Query `waitFor`
- `vi.hoisted()` required for mutable mock context shared across test cases
- CSRF `Origin` header required in POST/PATCH/DELETE/PUT route tests
- Zod v4: `parsed.error.issues[0]` (not `parsed.issues[0]`)
- URL parsing for nested routes: `.at(-2)` for `/{resource}/{id}/{action}` patterns
- Story closure requires Dev Agent Record checklist fully resolved
- EventBus from services only — never from API routes
- `mockReset()` for `Once` sequences — `clearAllMocks()` is NOT sufficient
- Explicit factory mocks always — never bare `vi.mock("@/db/queries/file")`
- New component with React Query dependency — update all affected test files immediately

### New from Epic 4:

- **Story spec navigation checklist** — every story that creates a new page/feature must specify navigation entry points (AppShell, BottomNav, AdminShell, Dashboard)
- **Route test mock pattern** — NEVER mock `withApiHandler` as passthrough. Mock `@/lib/rate-limiter` + `@/lib/request-context` dependencies instead. The real `withApiHandler` catches `ApiError` and returns correct HTTP status codes.
- **Dialog mock-to-null in jsdom** — CSS `md:hidden` doesn't apply in jsdom. Mock Dialog/DialogContent to return `null` when testing components with responsive desktop/mobile rendering.
- **`vi.hoisted()` for DB mocks** — when `vi.mock("@/db", ...)` references a `const` declared after the call, use `vi.hoisted()` to avoid TDZ errors.
- **Asymmetric server action returns** — document and follow the convention per `docs/decisions/server-action-returns.md` (AI-2).
- **`waitFor` for async rollback** — optimistic update rollback assertions must use `await waitFor(...)`, not synchronous assertions after `fireEvent.click`.
- **Dual feed function updates** — after AI-4 refactor, any `FeedPost` field change requires only ONE update to the shared select helper.
