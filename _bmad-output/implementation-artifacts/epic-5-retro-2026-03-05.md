## Challenges

1. **i18n hardcoding recurring despite team agreement** — Hardcoded English strings found in 5.3 review (channel tab component) and 5.4 review (ban error response). Pattern not documented in vi-patterns.ts. Team agreement alone is insufficient — needs checklist enforcement.

2. **Additive development creates invisible permission gaps** — The ban bypass (5.4 H1) was introduced when `banned` status was added in 5.4 but the join entry points (`joinOpenGroup`, `requestToJoinGroup`) predated the status and were not audited. New statuses/roles require entry-point regression tests.

3. **`successResponse` third-arg signature undocumented** — `successResponse(data, meta?, status=200)`: status is the _third_ argument. Hit mid-story in 5.3, required a route audit. Not in vi-patterns.ts.

4. **eventbus-bridge import cascade undocumented** — Adding any `@/db/queries/*` import to the bridge breaks `eventbus-bridge.test.ts` + `notification-flow.test.ts`. Workaround requires adding `vi.mock()` to both test files. Not documented.

5. **Permission bugs in multiple stories** — TOCTOU (read status outside transaction), visibility check ordering (joinType checked before visibility), `listGroupLeaders` dynamic import in event handler (5.2 H1–H3). Caught in review but reflect a gap in how permission-sensitive code is initially written.

6. **Runtime bugs surfaced in testing not caught by review** — Moderation bypass (posts go live even with `postingPermission = 'moderated'`) and @ autocomplete in channels not showing all members. Review process doesn't cover end-to-end runtime behavior.

7. **Open UX debt from 5.2** — `PendingRequestsSection` shows raw UUID values for pending member requests. Carried forward from 5.2 review.

## Key Technical Decisions & Lessons Learned

- **`successResponse(data, meta?, status)` — status is 3rd arg**: Use `successResponse({ x }, undefined, 201)` for 201 responses. Memorizing this wrong silently produces 200.
- **eventbus-bridge `@/db/queries/*` import cascade**: Any new query file import in the bridge requires `vi.mock("@/db/queries/new-file", () => ({ fn: vi.fn() }))` in both `eventbus-bridge.test.ts` and `notification-flow.test.ts`.
- **Additive development + permission status**: When a new member status or role is added, audit ALL entry-point functions (join, request, etc.) for gaps. Write regression tests at entry points, not just in the new status handler.
- **vi-patterns.ts is only useful if continuously updated**: Patterns added as retro action items but not updated mid-epic lead to rediscovery. The discipline is to add patterns during the story that discovered them.
- **Module-private helpers enforce correct architecture**: `FEED_SELECT_COLUMNS` and `_assemblePostPage` being module-private in `feed.ts` forced `getGroupFeedPosts` to live in the same file. Good constraint by design.

## Team Agreements

### Carried Forward from Epic 4 (still active):

- i18n keys defined in `en.json` + `ig.json` before any UI component is scaffolded
- `vi.useRealTimers()` required for tests involving React Query `waitFor`
- `vi.hoisted()` required for mutable mock context shared across test cases
- CSRF `Origin` header required in POST/PATCH/DELETE/PUT route tests
- Zod v4: `parsed.error.issues[0]` (not `parsed.issues[0]`)
- URL parsing for nested routes: `.at(-2)` for `/{resource}/{id}/{action}` patterns
- Story closure requires Dev Agent Record checklist fully resolved
- EventBus from services only — never from API routes
- `mockReset()` for `Once` sequences — `clearAllMocks()` is NOT sufficient
- Explicit factory mocks always — never bare `vi.mock("@/db/queries/file")`
- New component with React Query dependency — update all affected test files immediately
- Route test mock pattern — NEVER mock `withApiHandler` as passthrough
- Dialog mock-to-null in jsdom — CSS media queries don't apply
- `vi.hoisted()` for DB mocks — avoid TDZ errors
- Asymmetric server action returns — follow `docs/decisions/server-action-returns.md`
- `waitFor` for async rollback — optimistic update rollback assertions use `await waitFor(...)`

### New from Epic 5:

- **`successResponse` third arg** — `successResponse(data, meta?, status)` — status is 3rd; use `successResponse({ x }, undefined, 201)` for 201 responses
- **eventbus-bridge import guard** — any `@/db/queries/*` import in bridge requires `vi.mock()` in both `eventbus-bridge.test.ts` and `notification-flow.test.ts`
- **Additive permission audit** — when adding a new member status or role, audit ALL entry-point functions for gaps and add regression tests
- **vi-patterns.ts continuous updates** — add patterns during the story that discovered them, not only at retros
- **Pre-review checklist** — story template includes explicit i18n hardcoding check (after AI-4)
