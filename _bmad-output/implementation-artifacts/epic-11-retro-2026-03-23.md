## Challenges

1. **27 bugs found in manual testing — the most ever surfaced in a retrospective.** Discipline flow fundamentally broken from member perspective. Admin UI had systemic nav and color contrast issues. Moderation queue UX incomplete.

2. **Validation gap: tested internal correctness but never validated user-visible outcomes.** 391 unit tests verified mechanisms (database records created, sessions evicted, events emitted) but not what users actually see and experience. Suspended users saw "Something went wrong" instead of a suspension page.

3. **Discipline flow broken end-to-end from member perspective.** Admin suspends user → session evicted → but middleware redirect to `/suspended` not firing → member sees broken feed. No email notifications for any discipline action (warn, suspend, ban, content removal). Banned user not logged out, not shown ban message.

4. **Moderation queue UX incomplete.** Content column shows "—" for reported items (no content preview). Reporter identity hidden from admins. Reviewed items show same action buttons as pending (no status tags). No report abuse warning mechanism.

5. **Admin sidebar wiring gaps.** 6 admin pages built but nav links not connected in AdminSidebar. Each story added its page but didn't verify the sidebar was wired up.

6. **Dark theme color contrast issues across admin pages.** 6 instances of text/elements invisible against dark background. Tailwind classes not verified in actual admin theme context.

7. **Pre-existing test failures grew from 14 → 19.** 5 new failures in moderation.test.ts confirmed as pre-existing. Accumulating since Epic 8 without resolution.

8. **Epic 9's end-to-end verification lesson applied too narrowly.** The lesson was "multi-layer delivery chains need real environment verification." Applied correctly for notifications in Epic 10 (no chains), but missed the discipline delivery chain in Epic 11 (admin action → session eviction → middleware redirect → suspended page → email).

## Key Lessons Learned

### Lesson 1 — High test count does not equal validation

391 tests verified that the system _does_ things correctly (records created, sessions evicted, events emitted). Zero tests verified that users _experience_ things correctly (see the right page, receive the right email, can read the text on screen). Unit tests with mocks test mechanisms. User-perspective validation tests outcomes. Both are necessary. This is the single biggest lesson of the project so far.

### Lesson 2 — DoD must include user-perspective validation scenarios

Validation scenarios must be: (1) written at spec time by the story author, not improvised during testing, (2) described from the end-user perspective ("member sees suspension page with countdown"), not the system perspective ("database has status=suspended"), (3) walked manually with lightweight evidence (screenshot or one-line note) before marking "review," and (4) promoted to automated E2E for critical flows over time.

### Lesson 3 — Multi-layer chains need end-to-end walkthrough without exception

Any flow crossing system boundaries — middleware → redirect → page rendering, admin action → email delivery, keyword addition → retrospective content scan — requires manual verification of the complete chain. This extends Epic 9's notification delivery lesson to all cross-system flows.

### Lesson 4 — Admin UI needs baseline smoke test on every story

Every story that adds or modifies an admin page should include a 60-second visual check: all nav links resolve, all text readable against background, all interactive elements respond, reviewed states distinguishable from pending states. This catches the entire class of nav wiring and color contrast bugs.

### Lesson 5 — Architecture spike model continues to scale

The moderation spike (11 ADRs, DB schema, service boundaries, event contracts) prevented the kind of mid-epic rework seen in earlier epics. The pattern of "spike → implement → review" should remain standard for any epic introducing a new domain or cross-cutting system.
