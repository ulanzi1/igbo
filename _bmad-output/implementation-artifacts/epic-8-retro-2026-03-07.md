## Challenges

1. **Story 8.3 had the highest review severity of any Epic 8 story (3 HIGHs)** — The cross-cutting badge display task (7 display surfaces in a single task) created wiring verification gaps. `senderBadgeType` was never passed through `ChatWindow` to `MessageBubble` — chat badges were silently absent. The task size made it easy to miss individual surfaces.

2. **Redis cache deserialization type assumption broke** — `getUserBadgeWithCache` returned a string from `JSON.parse` but the type declared `Date`. No runtime validation existed on structured Redis cache values. The type system didn't catch it because the parse result was `any`.

3. **Misleading test file naming** — `VerificationBadge.integration.test.tsx` was named as an integration test but contained smoke/context tests that mocked child components. Someone reading the test file would assume badge rendering in real component trees was covered — it wasn't.

4. **Scope gaps identified late** — Two significant gaps discovered at retrospective time:
   - No member-facing "how do I earn points" information accessible from `/points`
   - No admin UI for gamification (rule editing, leaderboard, per-member investigation)
     These were not in any epic plan and were only surfaced during the retro discussion.

5. **Badge verification workflow undefined in PRD** — FR26 says "admins can assign verification badges to qualifying members" but no criteria, qualification process, or member-facing request workflow was ever planned. Discovered at retro time.

## Key Technical Decisions & Lessons Learned

- **Cross-cutting display tasks must be surface-specific subtasks.** When a component or data field must appear in N display locations, each location is its own numbered subtask with explicit wiring verification steps and a targeted render test. A single task covering 7 surfaces is a coverage gap waiting to happen.

- **Redis cache deserialization requires explicit type reconstruction.** `JSON.parse` returns `any`. Any structured Redis cache value containing `Date` fields must explicitly call `new Date(parsed.field)` on deserialization. Document this in `vi-patterns.ts` as a gotcha pattern.

- **`*.integration.test.*` filenames must mean integration.** Files named `integration.test` must exercise real component render trees. Smoke/context tests that mock children must use different naming (`*.smoke.test.*` or `*.context.test.*`).

- **`[REVIEW] validate with PO before ship` comments need a UI to act on.** The seeded points rule values (1/5/10 base_points, DAILY_CAP=100) were flagged for PO validation in 8.1 but there's no UI to experiment with them. Epic 11.6 (Gamification Rules Admin) is the resolution.

- **Pre-epic spikes deliver — repeat for every new infrastructure primitive.** Lua spike → 8.1 clean. Daily.co spike → 7.3 clean. This is now an established pattern, not an experiment.

- **Future phase: member-facing badge verification request flow.** Phase 2 will add a "Get Verified" entry point on the member profile page (post-registration, post-admin-approval). Admin review queue for badge requests will slot alongside Epic 11.8 (Member Points Investigation). Not planned for Epic 9.

## Scope Gaps Discovered & Resolved

### Gap 1 — `/points/how-to-earn` Page (Member-Facing)

**Issue:** Members can see their points balance and history but have no way to discover earning rules, badge levels/multipliers, or posting limit thresholds. The gamification system is opaque.

**Resolution:** Add one additional story to close out Epic 8's member-facing surface:

- **New Story: Points How-To-Earn Page** — Dedicated `/points/how-to-earn` page accessible from `/points`. Displays:
  - Earning rules (live from `platform_points_rules` — base_points per activity type)
  - Badge levels and multipliers (Blue 3x, Red 6x, Purple 10x)
  - Posting limit progression table (live from `platform_posting_limits`)
  - All data served from existing DB tables — no new schema
  - Page content reflects admin edits automatically (no hardcoded values)

### Gap 2 — Admin Gamification UI (Epic 11 Addition)

**Issue:** `platform_points_rules` and `platform_posting_limits` were spec'd as "admin-editable via future admin UI" but no such UI was assigned to any epic. Admins have no visibility into leaderboard or per-member points history.

**Resolution:** Add three stories to Epic 11 (Administration & Moderation):

| Story | Title                       | Scope                                                                                                                                           |
| ----- | --------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------- |
| 11.6  | Gamification Rules Admin    | Edit `platform_points_rules` base values; edit `platform_posting_limits` thresholds; validate seeded values (resolves `[REVIEW]` flag from 8.1) |
| 11.7  | Leaderboard Admin View      | Points ZSET top earners view; sortable/filterable; gaming pattern detection                                                                     |
| 11.8  | Member Points Investigation | Per-member points balance, ledger history, badge history, audit view; foundation for future badge request queue                                 |

### Gap 3 — Badge Verification Workflow (Planning Note)

**Issue:** FR26 defines badge assignment as admin-initiated but no criteria, qualification process, or discovery workflow was defined. Admin has no tooling to identify badge-eligible members.

**Current resolution:** Badge assignment remains purely admin-initiated (admin decides, uses admin UI). Epic 11.8 (Member Points Investigation) gives admins the data visibility to make informed badge decisions.

**Future phase (not Epic 9):** Add member-facing "Get Verified" button on profile page. Eligibility: registered + admin-approved members only. Creates a badge request that enters an admin review queue (slots alongside Epic 11.8 tooling). No timeline set.
