## Challenges

1. **`mockResolvedValueOnce` Bleed-Through — Hit Twice** — Stories 3.2 and 3.3 both hit the same bug: `vi.clearAllMocks()` clears call history but NOT the `Once` return queue. Tests using `mockResolvedValueOnce` sequences that end early contaminate subsequent tests. Fix (`mockReset()` in `beforeEach`) was documented after 3.2 and still needed re-discovery in 3.3. Root cause: muscle memory defaults to `clearAllMocks()`.

2. **`errorResponse` Signature Confusion (Story 3.3)** — Story spec showed `errorResponse(400, "msg", detail)` (3-arg form). Actual `@/lib/api-response` implementation takes a single `ProblemDetails` object. Developer had to discover this at runtime. Spec author and implementation were operating from different mental models — a documentation gap.

3. **Vitest Auto-Mock Cascade (Story 3.4)** — Bare `vi.mock("@/db/queries/follows")` caused Vitest to auto-import the module for structure analysis, loading transitive deps through `@/db` → all schemas → env cascade. Required explicit factory: `vi.mock("@/db/queries/follows", () => ({ ... }))`. Non-obvious failure mode with misleading error.

4. **`MemberCard` → `FollowButton` Cascade in Unrelated Tests (Story 3.4)** — Adding `FollowButton` to `MemberCard.tsx` broke `MemberGrid.test.tsx` (and other tests rendering `MemberCard`) because `FollowButton` needs `QueryClient`. Not obvious from reading `MemberCard.tsx`. Required retroactive mock addition to all affected test files.

5. **Accepted Technical Debt (3 items):**
   - `useFollow` doesn't invalidate `follow-list` queries after mutation — stale follower lists until background refetch
   - `FollowList` background refetch replaces accumulated "load more" pages
   - 20 parallel `GET /api/v1/members/[userId]/follow` requests per discover page (one per card)

## Key Technical Decisions & Lessons Learned

- **`mockReset()` not `clearAllMocks()`** for tests using `mockResolvedValueOnce` sequences
- **Explicit factory mocks** required for any query file with transitive DB imports (never bare `vi.mock("path")`)
- **Vitest auto-mock cascade** pattern: when mocking triggers unexpected imports, switch to explicit factory
- **`motion-safe:animate-*` + inline `style`** for animation delay (Tailwind purges dynamic class names at build time)
- **`localStorage` in `useState` initializer** needs SSR guard: `if (typeof window === "undefined") return true`
- **`computeFiltersForLevel` as module-level function** (not inside component body) for testability
- **`errorResponse` takes `ProblemDetails` object** — NOT `(status, title, detail)`. Use `throw new ApiError({ status, title, detail })` from `@/lib/api-error` for route errors
- **`db.transaction()` mock pattern**: `mockImplementation(async (cb) => cb(tx))` with tx object per test
- **Atomic count updates**: `onConflictDoNothing().returning()` check before incrementing; `GREATEST(..., 0)` floor on decrement

### AI-6 — Feed Algorithm Scope Decision _(Process — Epic 4 Story Quality)_

**Owner:** Alice (PO) + Winston (Architect)
**Status:** ✅ **Done** (2026-03-01)

**Decision:** Two-mode feed:

1. **Chronological** (default) — `ORDER BY created_at DESC` from followed members + groups + announcements
2. **Algorithmic** — `score = (recency_decay × 0.6) + (engagement_normalized × 0.4)` where `recency_decay` uses 12-hour half-life exponential decay and `engagement_normalized = (likes + 2×comments + 3×shares) / max_engagement_in_window` capped at 1.0. Weights defined as constants in `src/config/feed.ts`.

Cold-start behavior: < 50 total platform posts → chronological only; zero follows + zero groups → engagement-ranked platform-wide posts + prompt. Phase 2 additions (social graph signals, group relevance) deferred to > 1,000 active members.

Decision record at `docs/decisions/feed-algorithm.md` with exact AC language for Story 4.1.

**Success criteria met:** Algorithm defined with exact formula and constants — no room for interpretation.
