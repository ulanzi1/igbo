## Challenges

1. **Notification system not working end-to-end in real environment.** After Epic 9 was marked done, real-environment testing revealed: no in-app bell notifications, no email delivery, and `PushSubscriptionToggle` click doing nothing. Stories were verified at unit test level with mocked dependencies — the full delivery chain (event → router → preferences DB → channel → delivery) was never verified in a live environment.

2. **`NotificationRouter.route()` has no fallback on `getNotificationPreferences` failure.** Story 9.4 added a DB call inside the router with no try/catch. If the preferences table doesn't exist (migration not applied) or any DB error occurs, the exception propagates up through the EventBus handler, which swallows it silently. Result: all notifications go dark for that user with no visible error. A fragile critical path.

3. **Email handler wiring gap for mention/group_activity/post_interaction.** Story 9.4 added these types to user-configurable preferences (users can enable email), but the corresponding event handlers in `notification-service.ts` never had `emailData` wired (Story 9.2 only wired 3 handler types). Users who turn email ON in the matrix receive nothing because `deliverNotification()` skips email when `emailData === undefined`.

4. **UX gaps discovered post-ship from real product review.** Five UX issues found only after looking at the shipped `/settings/notifications` page: missing page padding, UTC timezone default (not device timezone), no post-save quiet hours summary state, confusing Push section ordering (after DnD), and duplicate push concept (matrix column + standalone toggle serving different purposes without explanation).

5. **`vi.hoisted()` TDZ requirement non-obvious.** Story 9.4 digest job tests required `vi.hoisted()` to avoid ES module TDZ errors when mock factories reference outer-scope variables. Non-obvious; took debugging time.

6. **`withApiHandler` dynamic params workaround repeated.** The pattern of extracting userId from URL path via `new URL(req.url).pathname.split("/").at(-N)` (because `withApiHandler` doesn't pass Next.js route params) was needed again in Story 9.4's dnd-status route. Established pattern but still a footgun for each new dynamic route.

## Key Technical Decisions & Lessons Learned

### Lesson 1 — Pipeline systems require end-to-end delivery verification, not just unit tests

For systems where data flows through 4+ layers (event emission → routing → channel decision → delivery), unit tests with mocked dependencies give false confidence. Each layer can pass individually while the chain is broken. For notification systems specifically, failures are **silent by design** — the EventBus swallows exceptions, the router returns early on suppression, email queues asynchronously. A broken chain is indistinguishable from a correctly-suppressed notification.

**The Definition of Done for pipeline/delivery stories must include:** at least one manually-verified delivery path in a real/dev environment before marking done.

### Lesson 2 — Critical path services must degrade gracefully, never go dark

Any service in the notification critical path that calls an external dependency (DB, Redis) must have a try/catch with documented fallback behavior. `NotificationRouter.route()` calling `getNotificationPreferences()` with no fallback is a single point of failure that blacks out all notifications. The correct behavior: catch errors, fall back to `DEFAULT_PREFERENCES`, log the error, continue delivery.

**Rule:** For any service in a delivery critical path — if the dependency fails, delivery should degrade to defaults, not stop entirely.

### Lesson 3 — Email handler wiring must be explicitly audited when eligibility changes

When `EMAIL_ELIGIBLE_TYPES` is modified (or DB preferences allow users to enable email for a type), there must be an explicit audit: does the event handler for that type pass `emailData`? The router saying "deliver email" and the handler not passing data are two silently-incompatible states. The audit result — wired or explicitly deferred — must be documented.

### Lesson 4 — UX review must evaluate the page as a unified experience, not isolated sections

Story 9.4's settings page was reviewed section by section (preferences matrix, quiet hours form, push toggle). No one asked "does the page work as a unified experience?" The Push column in the matrix and the standalone Push subscription toggle serve different purposes (per-type preference vs. device registration) — a distinction invisible to users. The page review process should include a holistic UX pass before calling a UI story done.
