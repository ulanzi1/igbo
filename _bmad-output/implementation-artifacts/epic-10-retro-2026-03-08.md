## What Went Well

1. **Relaxed, predictable delivery.** Dev (Project Lead) described Epic 10 as "comparatively relaxed" — a direct result of well-specified stories and a bounded domain. No post-ship follow-up story required.

2. **Clean layered architecture held across all three stories.** `search.ts` query layer → route → hook → component. Story 10.2 extended Story 10.1's foundation with filters and cursors without any rework. Story 10.3 added a new domain (recommendations) without touching the search layer.

3. **Epic 9 retro lessons applied without prompting.** Redis graceful degradation in Story 10.3 landed correctly from initial implementation — no review finding. Seven consecutive epics of full action item follow-through.

4. **Review process caught all HIGH issues before ship.** Zero post-ship functional bugs. Notable catches: `router.push` object form navigating to `[object Object]`, wrong DB column `start_time`, ARIA `aria-activedescendant` mismatch, blocked-user exclusion gap.

5. **Bilingual FTS delivered cleanly.** Full-text search across all 5 content types with 4-column bilingual coverage for articles (title, title_igbo, content, content_igbo). Conditional governance docs index forward-compatible with Epic 11 Story 11.5.

6. **Zero external credential complexity.** No new API keys, VAPID keys, or external services. Reduced integration risk compared to Epics 7–9.

## Challenges

1. **Story 10.2 had the highest review finding count (7, including 3 HIGH).** The story had the most intersecting concerns: cursor pagination, filter matrix, mode switching (overview vs filtered), URL sync, infinite scroll, type-specific cards, and highlight rendering — all in one story. High surface area correlates with higher review finding rate.

2. **`next-intl` router quirk not in `vi-patterns.ts`.** Story 10.2 F5: `useRouter().push()` from `next-intl` only accepts strings, not `{ pathname, query }` objects. This is a documented upstream behaviour difference from standard Next.js router. Not in shared knowledge base — available to trip up Epic 11.

3. **Widget placement spec-to-reality gap.** Story 10.3 `RecommendedGroupsWidget` on `/groups/page.tsx` was placed per spec ("above GroupList") but shipped without visual spacing between the widget and the "Search groups..." bar. Story spec gave placement instructions without a visual mockup. Caught by Dev in retro (cosmetic — one-line margin fix).

## Key Technical Decisions & Lessons Learned

### Lesson 1 — Well-specified stories in a bounded domain deliver predictably

Epic 10's delivery speed and quality were a direct product of: (1) stories that answered implementation questions upfront (cursor encoding strategy, filter-to-type applicability matrix, `ts_headline` sanitization requirement), and (2) a domain (search/recommendations) with a clear complexity ceiling and no cross-cutting side effects on other systems. This is the model to carry into Epic 11 story authoring.

### Lesson 2 — Story complexity ceiling correlates with review finding rate

Story 10.2 (7 findings, 3 HIGH) vs. Story 10.3 (5 findings, 1 HIGH) vs. Story 10.1 (6 findings, 2 HIGH). When a single story spans multiple architectural concerns simultaneously (pagination + filtering + mode switching + URL state + infinite scroll), review finding rate increases. Consider whether particularly wide stories warrant splitting or staged implementation.

### Lesson 3 — Widget placement specs need layout context, not just positioning instructions

"Add above GroupList" is not sufficient spec for a widget landing in a page with existing UI elements. Story specs for UI placement should reference a visual mockup or explicitly describe the spacing and visual relationship to adjacent elements. This prevents cosmetic gaps between spec and shipped result.

## Team Agreements

1. **Every admin action calls `logAdminAction()`** — no exceptions across all Epic 11 stories. Flag content, unflag content, suspend member, edit keyword, change rule config — all must produce an audit record.
2. **Moderation logic lives exclusively in `ModerationService`** — no moderation state, scanning, or flag lifecycle management scattered across routes or EventBus handlers.
3. **Moderation visibility default: Option B (visible-but-flagged)** — flagged content remains visible to all members; hidden only from non-admins on explicit moderator escalation. Prevents false-positive censorship of legitimate Igbo vocabulary.
4. **All moderation-related events follow the contract defined in `tech-spec-moderation-architecture.md`** — no per-story payload invention. Agreed shape from spike is authoritative.
5. **PR review check:** If an admin action exists in a story → reviewer verifies `logAdminAction()` call before approving.
