# Tech-Spec: AI-7 Bundled Implicit Requirements Fix

**Created:** 2026-04-05

## Overview

### Problem Statement

Portal Epic 1 retrospective (2026-04-05) identified that implicit requirements — i18n key usage, XSS sanitization, accessibility patterns, and component dependency awareness — account for roughly half of the ~66 review findings across 9 stories (~7.3/story). The rules exist in memory files, retro docs, and individual story reviews, but they are not encoded as either (a) pre-flight gates on story readiness, (b) dev completion gates, or (c) automated CI enforcement. Per Epic 0 Lesson 2 ("Discover → Standardize → Enforce → Reuse"), the team is strong at Discover and Standardize but has a persistent gap at Enforce. This gate-and-enforcement gap blocks Epic 2 start.

### Solution

Ship the bundled fix defined as AI-7 in `_bmad-output/implementation-artifacts/portal-epic-1-retro-2026-04-05.md`:

1. **Gate 1 (SM-enforced) — Story Readiness Checklist** embedded in the BMAD create-story template so every new story inherits mandatory i18n-key inventory, sanitization points, a11y patterns, and component dependency fields.
2. **Gate 2 (dev-enforced) — Dev Completion Checklist** augmenting the existing Definition of Done (SN-1) with the same categories, verified before PR.
3. **Playbook** — new `docs/monorepo-playbook.md` section "Frontend Safety & Readiness" capturing the rules Winston signs off on, linked from both checklists.
4. **CI enforcement** — two new scanners added to `scripts/ci-checks/` alongside the existing stale-imports / process-env / server-only scanners:
   - `check-hardcoded-jsx-strings.ts` — regex over `.tsx` files for hardcoded user-facing strings in JSX text nodes and known user-facing attributes.
   - `check-unsanitized-html.ts` — flags `dangerouslySetInnerHTML={{ __html: X }}` unless `X` contains a `sanitizeHtml(` call on the same expression or the file has an explicit allowlist comment.
5. **Real-enforcement proof (Lesson 2)** — each scanner ships with unit tests containing seeded fixture violations asserting the scanner actually reports them, so the success criterion "CI rule has failed at least once" is demonstrated deterministically and remains a regression test.

### Technical Decisions

1. **Regex over AST.** Match existing ci-checks convention. AST/ESLint would be more precise but introduces a new plugin + pipeline, fragments enforcement across two tools, and is out of scope per retro wording "CI rules for hardcoded strings in JSX + unsanitized dangerouslySetInnerHTML" — the retro already points at the existing ci-checks infrastructure.
2. **Hardcoded-string heuristic — multiline-aware with comment stripping (addresses F1, F7, F12, F13, F14, F16, F21).**
   - **Pre-process:** Before scanning each file, strip line comments (`// ...`), block comments (`/* ... */`), and JSX expression comments (`{/* ... */}`) by replacing them with same-length whitespace (preserves line/column offsets for error reporting). This eliminates the F2/F7 class of false-positives-from-comments and false-negatives-from-comment-hidden-strings.
   - **JSX text-node detection (F1 fix — multiline aware).** Do NOT use a single-line regex. Instead: scan the stripped source for `>...<` pairs using `/>([^<>{}]+)</gs` (dotall flag — matches across newlines). For each match, `trim()` the captured group. Reject (skip) if trimmed content is empty, starts with a digit, is purely punctuation, or does not contain at least one `[A-Za-z]` character. Then apply the user-facing heuristic: flag ONLY if the trimmed content matches `/[A-Za-z]/` AND (contains at least one space OR is ≥ 12 characters long) AND does not start with a lowercase JSX identifier pattern (avoids `<Link>postingTitle</Link>` false positive — bare identifiers rendered as children are conventionally lowercase camelCase or PascalCase single-word).
   - **Attribute detection (F13 fix — both quote styles, drop false-positive names).** Regex: `(title|alt|aria-label|placeholder)=(["'])([^"']{2,})\2`. Accepts both single and double quotes via backreference. `label` REMOVED from the list because `<option label>` and `<label htmlFor>` generate routine false positives without user-facing risk. Only flag if captured value contains a space OR is ≥ 12 characters.
   - **Exclusion list implementation (F21 fix — explicit sub-task, tests required).** Exclusions apply to the attribute detection only (text-node detection is already naturally scoped). Excluded attribute names: `className`, `id`, `role`, `type`, `name`, `key`, `href`, `src`, `htmlFor`, `data-*` (any), `aria-*` except `aria-label`, any `*-testid` / `*TestId`. These are compiled into the regex alternation and are tested in Sub 4.9.
   - **Template-literal attribute values (F13 partial).** Attribute values written as `placeholder={\`${foo} bar\`}` are intentionally NOT scanned (too dynamic to validate with regex). Documented explicit limitation.
   - **File scope (F12 decision).** `.tsx` only under `apps/**` and `packages/**`. Rationale: JSX by definition only appears in `.tsx` files; user-facing string constants in `.ts` service files that are later passed to components are a separate concern tracked as a follow-up, not this scanner's job. Acknowledged gap documented in scanner header comment.
   - **Skip files:** `*.test.tsx`, `*.stories.tsx`, `*.test.ts`, paths under `/e2e/`, `/test-fixtures/`. **Note (F22):** `/__tests__/` removed from skip list — the codebase uses co-located `*.test.tsx`, not `__tests__` directories; keeping the exclusion is dead code.
   - **i18n escape hatches (F7 fix — word-boundary, not substring).** A text-node match is skipped if the same JSX element source range contains a `{`-delimited expression that calls an i18n function. The detection uses word-boundary regex, NOT substring: `\b(useTranslations|t|Trans|useFormatter|formatMessage|intl)\s*\(` to prevent `start(` matching `t(` as substring. Additionally, the match is only suppressed if the i18n call is in the SAME JSX element as the text node, not merely anywhere on the line. Known remaining gap: a JSX element with both a literal text node AND a sibling expression calling `t(...)` will be suppressed — documented as acceptable false negative because such elements are rare and usually intentional.
   - **Allowlist comment:** `// ci-allow-literal-jsx` on the immediately-preceding line OR the same line as the match. Tested explicitly in both positions.

3. **Unsanitized HTML heuristic — strict prefix match + comment stripping (addresses F2, F16).**
   - **Pre-process identically to Decision 2.** Strip all comment syntaxes before scanning. This eliminates the "sanitizeHtml appears in a comment" false-negative (F2 case 2).
   - **Expression extraction (F16 fix — fail-closed).** For each `dangerouslySetInnerHTML={{` occurrence, locate the `__html:` key and extract the value expression using a brace/paren/bracket counter that is aware of string literals (single, double, backtick) and template-literal interpolation `${...}`. If the counter encounters a malformed state (unbalanced braces, EOF before close), **fail closed: report as a violation with reason `unsanitized-html-extraction-failed`**. Silent parse failures are precisely the Lesson 2 failure mode; we prefer false positives the dev must explicitly allowlist.
   - **Compliance rule (F2 fix — strict leading-call match).** After extracting and trimming the `__html:` expression, compliant iff the expression STARTS with `sanitizeHtml(` (i.e. matches `/^sanitizeHtml\s*\(/`). This blocks the short-circuit bypass `maybeSafe(x) || sanitizeHtml("")` because the leading token is `maybeSafe`, not `sanitizeHtml`. It also blocks `cond ? sanitized : raw` ternaries by design — if you need a conditional sanitized expression, you wrap the whole ternary in `sanitizeHtml(...)` at the outside, not at the inside. **The leading-call rule is a stricter compliance contract, documented in the Playbook.**
   - **Bare-reference still flagged.** `{{ __html: sanitizeHtml }}` (no parens) fails the leading-call match because `/^sanitizeHtml\s*\(/` requires the open paren — this was already an AC5 requirement and is preserved.
   - **Allowlist comment:** `// ci-allow-unsanitized-html` in the 3 lines immediately above the `dangerouslySetInnerHTML` occurrence. **Boundary tested (F23).** Rationale for 3 lines: accommodates a 1-line justification comment + a 1-line blank separator without being so generous that unrelated comments accidentally allowlist the site.
   - **Known limitation (revised per F2/F16).** The expression extractor is regex+counter, not a full TypeScript parser. It will conservatively fail-closed on exotic syntax (nested JSX in expression, complex ternaries with embedded object literals). **Revisit trigger:** if `unsanitized-html-extraction-failed` rate exceeds 3 across the repo OR any confirmed bypass is discovered, migrate to `@babel/parser` or `typescript` compiler API. Documented as scanner header comment.
4. **Allowlist comment format — standardized on EXISTING convention.** `// ci-allow-unsanitized-html` and `// ci-allow-literal-jsx`. The codebase already uses `// ci-allow-process-env` and `// ci-allow-no-server-only` for the existing scanners (verified in `apps/community/ci-checks.test.ts` during Step 2 investigation). Per Playbook §6.1 "Second Time = Standardize", new scanners MUST follow the same `// ci-allow-<reason>` format. Initial draft used colon-separated `ci-checks-allow: <reason>`; corrected during Step 2.
   - **Allowlist registry — merge-conflict-resistant format (F4 fix).** Aggregator `scripts/ci-checks/index.ts` regenerates `docs/ci-check-allowlist.md` on every run. **Registry rows contain `{file_path, reason, justification}` — NO line numbers.** Line numbers shift with every edit and would cause registry drift on every unrelated PR, generating merge conflicts 100% of the time between concurrent PRs. By keying on `(file_path, reason)` only, the registry is stable under line-drift and only conflicts when two PRs add the same reason to the same file (rare + legitimate review signal).
   - Rows sorted lexicographically by `(file_path, reason)` for deterministic output.
   - Registry includes ALL existing allowlist comments (`ci-allow-process-env`, `ci-allow-no-server-only`) plus the two new ones, so the audit surface is unified.
   - **Justification extraction:** the comment on the line IMMEDIATELY BEFORE the `// ci-allow-*` marker (or the suffix of the marker line if no prior-line comment exists). Devs adding an allowlist entry are expected to document the reason on the line above — Playbook §7 will codify this.
   - **Drift detection:** if on-disk `docs/ci-check-allowlist.md` differs from generated content (byte-for-byte after normalization), aggregator exits non-zero with `check: "allowlist-registry-drift"`.
   - **Self-heal on local run:** developers running `pnpm ci-checks` locally get the regenerated file written in place, so re-running and committing fixes the drift. **The registry is scanner-adjacent metadata, NOT a new scanner class.**
5. **SN-5 Readiness Checklist.** Added as a new numbered gate section (SN-5) so it composes with existing SN-1..SN-4 nomenclature. Explicitly marked "Gate 1 — Story cannot enter development if any box is unchecked" per retro Team Agreement.
6. **Dev Completion additions live in SN-1.** Rather than creating SN-6, append 4 bullet points to the existing DoD to avoid fragmenting completion criteria across two sections.
7. **Test file location — single authoritative surface (F8 fix).** **All scanner tests live in `apps/community/ci-checks.test.ts`** — exhaustive + canary in one file. **Co-located `scripts/ci-checks/check-*.test.ts` files are NOT created.** Earlier drafts proposed a "dual layer" with co-located exhaustive suites + wrapper smoke coverage; the adversarial review correctly identified that the co-located files are not picked up by any workspace test config and would rot silently. Single surface eliminates dual-maintenance drift and guarantees every test runs in CI. The wrapper is extended with TWO new describes (`scanHardcodedJsxStrings`, `scanUnsanitizedHtml`), each containing the full exhaustive test set (seeded violations, compliant fixtures, allowlist handling, empty-tree edge cases, VD-5 cross-path, Lesson 2 bare-identifier, adversarial regex cases).
8. **Extend existing wrapper, do NOT create new.** `apps/community/ci-checks.test.ts` already has three describes (`scanForStaleImports`, `scanDirectProcessEnv`, `scanMissingServerOnly`) + an integration canary at the bottom asserting zero violations against `ROOT`. This spec adds the two new describes immediately before the canary block and updates the canary's combined spread to include both new scanners. Creating a parallel `ci-frontend-safety-scanners.test.ts` would fragment the CI test surface and duplicate the canary — rejected.
9. **No in-process memoization of `collectTsFiles` (F9 fix).** The earlier "memoize if canary > 5s" proposal is withdrawn. Module-level caches break tmpdir fixture tests (subsequent tests see stale results). If the canary is slow, the acceptable mitigations are (a) accept 5–10s as a reasonable CI cost for the zero-grandfathering guarantee, (b) pass a per-invocation `Map<string, string[]>` cache object threaded as an explicit argument through `collectTsFiles(rootDir, { cache })`, never module-level. Tasks reflect (a) as the default — measurement only, no caching unless the team later decides the runtime is unacceptable.
10. **Kill-switch fails closed in CI (F5 fix).** Aggregator reads `CI_CHECKS_DISABLE` ONLY when `process.env.CI !== "true"` (GitHub Actions and most CI providers set `CI=true` automatically). In CI, the env var is ignored with a warning banner. Locally, the env var skips the named scanner classes with an explicit banner output. This structurally prevents the kill-switch from becoming a CI bypass path. Documented in Playbook §7.
11. **Evidence pinning by commit SHA (F30 fix).** The live-CI-validation evidence (screenshot + GitHub Actions run URL) must include the commit SHA of BOTH the spec branch HEAD AND the validation branch HEAD at the time CI ran. Task 11 Sub 11.5 records these SHAs in the story Validation Evidence section. Any force-push to the spec branch after evidence capture requires re-running validation.
