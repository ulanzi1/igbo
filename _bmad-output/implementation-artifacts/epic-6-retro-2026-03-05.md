## Challenges

1. **i18n hardcoding in every single story — despite two prior action items** — Found at HIGH severity in 6.1 (H3: mention-suggestion + meta form), 6.2 (H-1: 18 hardcoded strings), 6.3 (H1: bilingualBadge + category). The checklist is being checked but strings are hardcoded _during initial scaffolding_, before the checklist runs. This is a **construction problem, not an inspection problem**.

2. **Missing mutation/action tests** — `getArticleForEditing` tests missing in 6.1 (M2); `ArticleReviewActions.test.tsx` missing entirely in 6.2 (H-4). Read paths are well-tested; mutation side effects (toasts, query invalidation, API calls from action components) are consistently under-tested.

3. **`dangerouslySetInnerHTML` XSS risk** — 6.2 preview modal rendered user-generated HTML without sanitization. `sanitize-html` was already installed and used server-side; it wasn't applied before client rendering. Needs explicit documented pattern.

4. **`BROWSE` rate limit preset referenced but doesn't exist** — Story 6.3 spec suggested using `BROWSE` rate limit preset; preset doesn't exist in `src/services/rate-limiter.ts`. Caught at runtime (not build time). Story specs should only reference presets that are verified to exist.

5. **Feature gaps discovered post-implementation** — After Epic 6 stories were complete, Dev identified missing author experience features: no My Articles dashboard, no article revision flow (only hard reject), no email notifications, no visual feedback in Tiptap editor, cover image not mandatory. These are captured as Story 6.4.

## Key Technical Decisions & Lessons Learned

- **i18n is a construction problem**: Checklist enforcement is insufficient. Keys must be defined _as part of Task 1_ (alongside DB schema) so the dev agent has them from line one — not as a cleanup task at the end. The root cause is that hardcoding happens during component scaffolding, not after.
- **Spikes deliver**: AI-5 (ISR pattern) → 6.3 implemented correctly first try. AI-6 (bilingual editor) → 6.1 implemented correctly first try. The investment in spikes before story creation is working.
- **`dangerouslySetInnerHTML` safe-use rule**: Any user-generated HTML rendered via `dangerouslySetInnerHTML` MUST be sanitized server-side with `sanitize-html` before passing to the client component. The sanitized HTML is safe to use client-side.
- **Mutation test gap**: Component tests tend to cover render/read paths well but miss mutation paths (form submit, API call, toast, query invalidation). Explicitly required in story test tasks.
- **Rate limit preset hygiene**: Story specs must only reference rate limit presets that are confirmed to exist in `src/services/rate-limiter.ts`. Add a `BROWSE` preset or remove it from story template suggestions.

## Team Agreements

### Carried Forward (still active):

- i18n keys defined in `en.json` + `ig.json` before any UI component is scaffolded
- `mockReset()` in `beforeEach` — never `clearAllMocks()`
- Explicit factory mocks always — never bare `vi.mock("@/db/queries/file")`
- CSRF `Origin` header required in all mutating route tests
- Zod v4: `parsed.error.issues[0]` (not `parsed.issues[0]`)
- `successResponse(data, meta?, status)` — status is 3rd arg; `successResponse({ x }, undefined, 201)` for 201s
- eventbus-bridge import guard — new `@/db/queries/*` import in bridge requires `vi.mock()` in both integration test files
- `mockReset()` for `Once` sequences — `clearAllMocks()` is NOT sufficient
- Route test mock pattern — NEVER mock `withApiHandler` as passthrough
- `vi.hoisted()` for DB mocks — avoid TDZ errors
- No `auth()` in ISR Server Components — auth-gated features use `useSession()` in Client Components
- `dangerouslySetInnerHTML` requires prior `sanitize-html` sanitization (new — from 6.2 XSS finding)

### New from Epic 6:

- **i18n keys in Task 1** — namespace and key names defined alongside DB schema in story spec; not deferred to late-stage i18n task
- **Mutation test coverage** — story test task must explicitly require tests for action components (submit, approve, reject, feature toggle) — not just render/read paths
- **Rate limit preset hygiene** — story specs only reference presets confirmed to exist in `src/services/rate-limiter.ts`
- **`dangerouslySetInnerHTML` safe-use** — server-side `sanitize-html` required before any client-side `dangerouslySetInnerHTML` usage
