## Challenges

1. **Build-Test-Skip-Validate anti-pattern.** Mechanisms were tested in isolation (unit tests with mocks verify config structure, script syntax, component rendering) but wiring between mechanisms was never tested. Load test scripts were validated syntactically but never executed. Backup scripts were tested for correct flag generation but never run against a database. Monitoring dashboards were tested as JSON but never verified in Grafana.

2. **PITR non-functional despite passing tests.** `restore-pitr.sh` uses `pg_dump` (logical backup) but WAL replay requires `pg_basebackup` (physical backup). Tests verified the script generates correct flags — but the approach itself is architecturally wrong. Review caught it; development didn't.

3. **Spec compliance gaps recurred.** Every story had review findings where the spec required something and the implementation missed it. Not bugs in logic — gaps in coverage. The spec says "do X, Y, Z" and implementation delivers X and Y but misses Z. Pattern repeated across all 7 stories.

4. **No end-to-end flow verification.** No story verified that its deliverables work in a real environment. Docker Compose was never `docker compose up`-ed. k6 scripts were never run against a server. Backup scripts were never executed against PostgreSQL. Sentry was never verified to receive an actual error. Every component is individually correct but collectively unproven.

5. **Epic 11 action items not followed through.** 0/4 action items fully completed. Validation scenarios not added to specs. Walkthrough evidence not required. Admin smoke checklist not created. The same gap identified in Epic 11 persisted through Epic 12 because the fixes were process reminders, not structural gates.

## Key Lessons Learned

### Lesson 1 — The root cause: optimizing for local correctness instead of system correctness

The project optimized for making each component individually correct (and it succeeded — 4,795 tests prove it). But nobody owned the question "does this feature work for a user?" Mock-heavy unit tests validate that components do what they're told. They don't validate that components are told to do the right thing, in the right order, connected to the right dependencies.

This is Pattern D: **No ownership of end-to-end flow.** Everyone owns their layer. Nobody owns "this works."

### Lesson 2 — Guidelines don't survive delivery pressure. Gates do.

Epic 11 produced 4 action items. All were process guidelines ("add a section to the template," "require evidence before review"). None had enforcement mechanisms. When delivery pressure arrived in Epic 12, all were skipped because nothing prevented shipping without them.

**The fix is structural:** Make validation a gate, not a guideline. If validation scenarios aren't in the spec, the story can't start. If walkthrough evidence isn't in the story file, the story can't move to review. If the flow isn't proven, it's not done.

### Lesson 3 — "Technically complete" is not "production-ready"

Every story in Epic 12 is done. Every test passes. Every review finding is fixed. But the platform has never been run as a complete system. Docker Compose has never been started. Load tests have never been executed. Backups have never been restored. Monitoring has never received a real error.

**Production-ready = evidence that the platform works, survives, and recovers under real conditions.** Not "all stories done."

### Lesson 4 — The test pyramid has a missing middle

The project has 4,795 unit tests (strong base) and ~3 E2E tests (thin top). The middle layer — integration tests that verify real connections between components without full E2E overhead — doesn't exist. This is where "the backup script connects to the real database" and "the monitoring middleware sends metrics to the real Prometheus endpoint" would live.

### Lesson 5 — Review is the only safety net, and that's not sustainable

The adversarial review model (claude-sonnet-4-6 implements, claude-opus-4-6 reviews) caught 54 real issues in Epic 12 alone. But review is a detection mechanism, not a prevention mechanism. Issues should be caught during development through validation, not discovered during review. The review model is excellent — but it shouldn't be the only line of defense.

## Launch Gates

Dev (Project Lead) defined four hard launch gates. Nothing else matters until they pass.

### LG-1 — E2E User Flow Proven

**Gate:** A real user flow — signup → verify email → login → create post → receive notification → view in feed — works end-to-end in a deployed environment.
**Evidence required:** Demonstrated flow, not test assertions.
**Owner:** Charlie (Senior Dev)

### LG-2 — Load Test Executed

**Gate:** k6 scripts run against a real server. Seeder populates real data. Results documented with actual numbers (requests/sec, p95 latency, error rate).
**Evidence required:** k6 output report with real metrics.
**Owner:** Winston (Architect)

### LG-3 — Monitoring Verified

**Gate:** Trigger a real error → verify it appears in Sentry. Generate real traffic → verify metrics appear in Prometheus/Grafana. Verify at least one alert rule fires.
**Evidence required:** Screenshots or logs showing real data flowing through monitoring.
**Owner:** Charlie (Senior Dev) + Elena (Junior Dev)

### LG-4 — Backup/Restore Executed

**Gate:** Run a daily backup against real PostgreSQL. Restore the backup to a fresh instance. Verify data integrity (row counts, sample queries).
**Evidence required:** Documented backup/restore with verification queries.
**Owner:** Winston (Architect)

### LG-5 — PITR Risk Acceptance

**Gate:** Alice (PO) explicitly accepts the risk: daily backup provides 24h RPO; PITR is non-functional.
**Conditions:** (1) Alice acknowledges and accepts 24h RPO for launch, (2) platform is early-stage with low initial data criticality, (3) PITR fix is scheduled as first post-launch task.
**Owner:** Alice (Product Owner)

## Structural Non-Negotiables

These are not guidelines. They are gates — enforced structurally, not by reminder.

### SN-1 — Redefine "Done" = Flow Verified

A story is not done when all tasks are complete and tests pass. A story is done when the feature works for a user in a real or realistic environment. The definition of "done" must include proof of flow, not just proof of components.

### SN-2 — Executable Validation Scenarios

Validation scenarios must be written at spec time, described from the end-user perspective, and verified with evidence before a story can move to review. They are not optional addenda — they are part of the spec. If a story doesn't have validation scenarios, it's not ready for development.

### SN-3 — Integration Layer Testing (The Missing Middle)

The project needs tests that verify real connections between components — not just unit tests with mocks and not full E2E. Tests like "the backup script connects to a real database" and "the WebSocket reconnection works against a real server" fill the gap between 4,795 unit tests and 3 E2E tests.

### SN-4 — Flow Ownership

Every feature has an owner who is responsible for "this works for a user," not just "my layer works." Flow ownership means someone verifies the complete chain — not just their component.

## Technical Debt

### TD-1 — PITR Non-Functional (HIGH)

`restore-pitr.sh` uses `pg_dump` (logical) but WAL replay requires `pg_basebackup` (physical). Fix: implement proper `pg_basebackup` + WAL archiving for PITR, or explicitly document and accept daily-backup-only RPO.

### TD-2 — Load Test Scripts Never Executed (MEDIUM)

k6 HTTP and WebSocket scripts exist but have never run against real infrastructure. Unknown bugs may exist in pagination logic, WebSocket handshake, or seeder data relationships.

### TD-3 — Maintenance Page Nested HTML (LOW)

503 maintenance page uses nested `<html>` within Next.js error boundary. Works but is technically invalid HTML. Should be a standalone static page served by reverse proxy.

### TD-4 — Seeder Visibility Distribution (LOW)

Synthetic seeder creates all groups as `public`. Should include `private` and `hidden` groups for realistic load test scenarios.
