## Successes & Strengths

### 1. Progressive Implementation Created Compounding Quality

The sequential story flow — scaffolding → security → design system → layout → features — meant every new piece got validated against what already existed. Code review findings in early stories became established habits in later ones:

- i18n-first development stabilized by Story 1.6
- Soft-delete filter discipline became automatic by Story 1.10
- Test mock patterns standardized by Story 1.12
- Security patterns (timing attack prevention, constraint violation handling) learned progressively

### 2. Architectural Decisions Held Stable

Phase A decisions proved durable through all 17 stories without revision:

- `withApiHandler()` wrapper — used in every REST route
- RFC 7807 Problem Details — consistent error format
- EventBus service isolation — services never call each other directly
- Soft-delete pattern — GDPR-compliant from the start
- Two-container architecture (Next.js + Socket.IO) — clean separation
- Redis sorted set rate limiting — horizontally scalable

### 3. Code Review Discipline Caught Critical Issues

100+ findings across the epic, including security-critical catches:

- Timing attacks in login flow (Story 1.7)
- OAuth secret exposed in GET URL parameter (Story 1.9)
- `passwordHash` not cleared during anonymization (Story 1.13)
- Magic byte check not comparing declared vs detected MIME (Story 1.14)
- Rate limiter using passive `socket.onAny()` instead of blocking `socket.use()` (Story 1.15)
- Cancel-deletion route with no rate limiting (Story 1.13)

### 4. Test Suite Growth

From 3 tests (Story 1.1a) to 973 (Story 1.17), all passing. Standardized patterns:

- Co-located tests (no `__tests__/` directories)
- `vi.mock()` patterns for next-intl, next-auth, ioredis, @/env
- `vi.hoisted()` for mock factory TDZ issues
- `@vitest-environment node` for server files, `jsdom` for components

## Challenges & Growth Areas

### 1. `drizzle-kit generate` Broken Across Entire Epic

Every migration (0001–0012) had to be hand-written SQL because `drizzle-kit generate` fails with `server-only` import errors. Works but is slower and more error-prone than auto-generated migrations.

### 2. i18n Hardcoding Was the #1 Recurring Review Finding

Hardcoded English strings caught in code review for the first 6 stories (1.3–1.8): button labels, form placeholders, error messages, sidebar labels. Pattern stabilized by Story 1.6 but represented avoidable rework.

### 3. Soft-Delete Filters Missed in 8 Queries

`.where(isNull(deletedAt))` was initially missing in new queries across Stories 1.5, 1.6, 1.8, and 1.9. Code review caught all instances, but a default query filter or helper function could prevent this systemically.

### 4. Test Mock Cascade Failures

Adding features in one component broke tests in unrelated files:

- `LanguageToggle` using `useSession()` broke 4 layout test files (Story 1.11)
- Rate limiting in `withApiHandler()` required mocking in every route test (Story 1.12)
- `@/env` import cascade forced mocking in any test touching modules that import it

### 5. Story 1.15 Was Oversized

10 HIGH findings, 11 MEDIUM findings, 136 new tests — nearly double the next largest story. Combined Socket.IO server, Redis adapter, two namespaces, notification service, block/mute tables, REST API, and SocketProvider. Should have been split into 2-3 stories.

### 6. No UI Verification Step in Code Review

Code review focuses on technical quality but doesn't provide a summary for the project lead to verify visual output. Missing: routes to visit, components to inspect, expected visual states.

### 7. Infrastructure Stories Lacked Visual Feedback

Story 1.15's realtime infrastructure couldn't be visually validated until future epics generate real events. No dev-only test harness was provided for immediate verification.

### 8. Architecture Doc vs Epics Doc Conflict

`architecture.md` lists three Socket.IO namespaces (`/chat`, `/notifications`, `/events`), but `epics.md` (authoritative) specifies two. Caused confusion during Story 1.15 implementation.

## Key Insights & Lessons Learned

### Technical Patterns Established

| Pattern         | Details                                                                                   |
| --------------- | ----------------------------------------------------------------------------------------- |
| Migrations      | Hand-write all SQL; `drizzle-kit generate` fails with `server-only` error                 |
| Zod             | Import from `"zod/v4"`, use `.issues[0]` not `.errors[0]`                                 |
| API Routes      | `withApiHandler()` wrapper, RFC 7807 via `successResponse()`/`errorResponse()`            |
| Admin Routes    | `requireAdminSession()` from `@/lib/admin-auth.ts`                                        |
| User Routes     | `requireAuthenticatedSession()` from `@/services/permissions.ts`                          |
| Auth Import     | `@/server/auth/config` (NOT `@/auth`)                                                     |
| Rate Limiting   | Dynamic `await import("@/lib/rate-limiter")` in `withApiHandler()` to avoid test cascades |
| Services        | `import "server-only"` first line; emit via EventBus, never from routes                   |
| i18n            | `useTranslations("Namespace")` — no hardcoded strings anywhere                            |
| Tests           | Co-located; `vi.hoisted()` for mock TDZ; regular `function()` for `new`-called mocks      |
| Barrel Exports  | All public API via `src/features/x/index.ts`; ESLint enforces                             |
| Email Templates | Pure functions, no env imports; callers pass full URLs as data                            |
| EventBus        | Emit from services only; synchronous; `domain.action` naming with underscores             |
| Session         | `getCachedSession()` for realtime auth; batch eviction on tier change                     |

### Process Lessons

1. **i18n-first development** — deploy i18n keys before writing UI, not after
2. **Security by review** — code review is the primary security gate; every critical gap was caught
3. **Error enumeration prevention** — login, password reset, application checks return uniform error messages
4. **Transaction scope** — password reset requires atomic: update password + mark token + delete sessions
5. **Edge runtime awareness** — `next-auth/jwt` decode is Edge-compatible; full auth config is not
